# `dofbot_moveit` — MoveIt 2 Motion Planning Configuration

> **Type:** ament_cmake config-only package  
> **Role:** MoveIt 2 configuration — defines planning groups, kinematics solver, controllers, SRDF, and launch files for the 4-DOF Dofbot arm.

---

## Purpose

This package was generated by the **MoveIt Setup Assistant** and then customized. It provides everything MoveIt 2 needs to plan and execute trajectories on the physical hardware:

- SRDF (Semantic Robot Description Format) — planning groups, end effectors, passive joints
- ros2_control controller definitions — `arm_controller`, `gpio_controller`
- ros2_control hardware plugin binding (xacro) — loads `dofbot_hardware/DofbotSystemHardware`
- Kinematics solver config — KDL
- Launch files — hardware bringup, MoveIt move_group, RViz

## `ros2_controllers.yaml` — Controller Definitions

```yaml
controller_manager:
  ros__parameters:
    update_rate: 20  # Hz — matches ESP32 polling rate

    arm_controller:
      type: joint_trajectory_controller/JointTrajectoryController
    joint_state_broadcaster:
      type: joint_state_broadcaster/JointStateBroadcaster
    gpio_controller:
      type: forward_command_controller/MultiInterfaceForwardCommandController

arm_controller:
  ros__parameters:
    joints: [arm1_Joint, arm2_Joint, arm3_Joint, arm4_Joint]
    command_interfaces: [position]
    state_interfaces: [position]

gpio_controller:
  ros__parameters:
    joint: aux_hardware
    interface_names: [led_r, led_g, led_b, torque_enable, buzzer_trigger]
```

### Controller Roles

| Controller | Type | What It Does |
|------------|------|-------------|
| `arm_controller` | JointTrajectoryController | Receives planned trajectories from MoveIt, interpolates, writes position commands to `dofbot_hardware` |
| `joint_state_broadcaster` | JointStateBroadcaster | Reads `hw_states_[]` from hardware, publishes `/joint_states` topic |
| `gpio_controller` | MultiInterfaceForwardCommandController | Receives `Float64MultiArray` on `/gpio_controller/commands`, writes directly to `aux_hardware` interfaces |

## `dofbot_urdf.ros2_control.xacro` — Hardware Plugin Binding

This xacro macro injects the `<ros2_control>` block into the URDF at build time:

```xml
<ros2_control name="${name}" type="system">
    <hardware>
        <plugin>dofbot_hardware/DofbotSystemHardware</plugin>
    </hardware>
    
    <!-- 4 position-controlled joints -->
    <joint name="arm1_Joint"> ... </joint>
    <joint name="arm2_Joint"> ... </joint>
    <joint name="arm3_Joint"> ... </joint>
    <joint name="arm4_Joint"> ... </joint>

    <!-- GPIO peripheral interfaces -->
    <gpio name="aux_hardware">
        <command_interface name="led_r"/>
        <command_interface name="led_g"/>
        <command_interface name="led_b"/>
        <command_interface name="torque_enable"/>
        <command_interface name="buzzer_trigger"/>
    </gpio>
</ros2_control>
```

This tells the Controller Manager to load `dofbot_hardware/DofbotSystemHardware` and expose exactly these joint + GPIO interfaces.

## `dofbot_urdf.srdf` — Semantic Robot Description

### Planning Groups

| Group | Joints/Links | Purpose |
|-------|-------------|---------|
| `dofbot_arm` | `virtual_joint`, `arm1-4_Joint`, `Camera_Joint` | Primary planning group (4 active DOF) |
| `hand` | `arm5_Link`, gripper linkage links | End effector group |

### Named Poses (Group States)

| Pose Name | arm1 | arm2 | arm3 | arm4 | Notes |
|-----------|------|------|------|------|-------|
| `zero` | 0° | 0° | 0° | 0° | All joints straight |
| `ready` | 0° | -42.2° | 42.2° | 55.2° | Bent working pose |
| `home` | 90° | -90° | 90° | 90° | Full range test |

### Passive Joints (mimic joints — not planned)
- `Llink1_Joint`, `Llink2_Joint`, `Llink3_Joint`
- `Rlink1_Joint`, `Rlink2_Joint`, `Rlink3_Joint`

These are the parallel gripper linkage joints that mimic `Rlink1_Joint`.

### End Effector
- **Name:** `fingers`
- **Parent link:** `Camera_Link`
- **Group:** `hand`

### Virtual Joint
- **Name:** `virtual_joint`
- **Type:** `fixed`
- **Parent frame:** `world` → **Child link:** `base_link`

## `moveit_controllers.yaml` — MoveIt Controller Bridge

```yaml
moveit_controller_manager: moveit_simple_controller_manager/MoveItSimpleControllerManager

moveit_simple_controller_manager:
  controller_names: [arm_controller]
  arm_controller:
    action_ns: follow_joint_trajectory
    type: FollowJointTrajectory
    default: true
    joints: [arm1_Joint, arm2_Joint, arm3_Joint, arm4_Joint]
```

This tells MoveIt's `MoveGroupNode` to send planned trajectories to `arm_controller` via the `FollowJointTrajectory` action interface.

## `robot_bringup.launch.py` — Master Launch File

This is the primary launch file that starts the entire hardware + control stack:

```
t=0s   → hardware_test.launch.py
         ├── Robot State Publisher (parses URDF via MoveItConfigsBuilder)
         └── ros2_control_node (Controller Manager + dofbot_hardware plugin)
t=0s   → static_virtual_joint_tfs.launch.py (world → base_link TF)
t=2s   → joint_state_broadcaster spawner
t=4s   → arm_controller spawner
t=6s   → move_group.launch.py (MoveIt MoveGroup node)
t=8s   → moveit_rviz.launch.py (RViz with MoveIt panel)
t=10s  → gpio_controller spawner
```

The **staggered TimerAction** boot sequence ensures each component is ready before the next one starts.

### `hardware_test.launch.py` — Base Layer

Uses `MoveItConfigsBuilder` to:
1. Parse the URDF (resolves xacro, injects ros2_control block)
2. Start `robot_state_publisher` with the resolved URDF
3. Start `ros2_control_node` (Controller Manager) with the URDF + `ros2_controllers.yaml`

## How MoveIt Connects to Hardware

```
MoveIt MoveGroup
    │ /plan_kinematic_path service    (used by AI agent)
    │ /execute_trajectory action      (used by AI agent)
    │
    ▼
MoveIt Simple Controller Manager
    │ FollowJointTrajectory action
    │
    ▼
arm_controller (JointTrajectoryController)
    │ writes position commands
    │
    ▼
dofbot_hardware::DofbotSystemHardware
    │ USB serial
    ▼
ESP32 → Servos
```
